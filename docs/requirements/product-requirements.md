# 多人物传记时间轴系统 — 产品需求文档

## 1. 项目概述

构建一个支持多人物传记的时间轴可视化与知识管理系统。用户可以上传人物传记文本，系统自动提取人物、事件、时间信息，处理时间点/时间段的混合表示，并能够在多人传记之间对齐共享事件。前端以泳道时间轴进行可视化展示，系统分成管理模块（数据生产）与展示模块（叙事阅读）两大部分。

## 2. 核心概念与数据结构

### 2.1 时间表达的统一模型

传记中的时间既有精确时间点（"1949年10月1日"），又有时间段（"1949-1956年"）和模糊时段（"少年时代"）。

方案：用字段组合覆盖所有形态，将"原文描述"与"可计算机时间"分离。

- 时间段：`start_date` ≠ `end_date`
- 时间点：`start_date` = `end_date`
- 模糊时段：用近似范围存储，标记 `granularity`

### 2.2 多人物共享事件模型

从单人时间轴扩展到多人，将事件提升为独立实体，通过关联表承载每个人物的视角。

三张核心表：`person`（人物）、`event`（标准事件）、`person_event`（人物-事件关联）

### 2.3 人物别名与外部信息扩展

支持人物识别与重名处理：`person_alias` 表存储别名映射，`external_person_info` 表存储自动采集的外部信息。

## 3. 技术栈

| 层级 | 技术 |
|------|------|
| 后端语言 | Python 3.11+ |
| 后端框架 | FastAPI |
| 数据库 | PostgreSQL 15+ |
| 文本处理 | PyMuPDF（PDF 文本提取） |
| NER/事件抽取 | spaCy / HanLP + 大语言模型辅助（OpenAI API、Qwen 等） |
| 时间解析 | dateparser、ctparse、自定义规则 |
| 实体对齐 | sentence-transformers（多语言模型） |
| 前端框架 | Vue 3 (Composition API) + Vite |
| 状态管理 | Pinia |
| 可视化 | D3.js (SVG) |
| 样式 | Tailwind CSS v4 + Headless UI |
| 前端路由 | Vue Router（管理端 /admin，展示端 /view） |
| 网络请求 | Axios / fetch |
| 包管理 | pnpm |

> 注：第一阶段暂不引入 Celery/Redis 任务队列和 OCR 流水线，直接同步处理。OCR 能力后续阶段再集成。

## 4. 事件管理策略

- **手动创建是基础能力**，用户可完全脱离 AI 使用系统
- AI 自动抽取是增强功能，抽取结果需人工审核确认
- 手动 CRUD 与 AI 抽取并存

## 5. 页面结构

### 5.1 管理端（/admin）

- 人物管理列表：姓名、生卒年、事件数量、状态、操作按钮
- 人物详情/编辑页
- 事件库：按人物、类型、时间搜索
- 事件创建/编辑页（手动）
- 对齐工作台：待确认的事件对齐对，并排对比、一键合并/拆分
- 上传与处理页：上传传记文件，查看处理进度

### 5.2 展示端（/view）

- 多人物泳道时间轴（Git 图风格）
- 单人物时间轴视图
- 事件详情浮层卡片
- 时间缩放、聚焦模式

## 6. 数据处理管道

### 阶段 1：文本提取
- TXT 直接读，PDF 用 PyMuPDF
- 原始文本存入 `biography_text` 表

### 阶段 2：NER 与指代消解
- spaCy/HanLP + 历史人物词表增强
- 通过 `person_alias` 表映射到标准 `person_id`

### 阶段 3：事件抽取
- 大模型结构化抽取，返回 JSON
- 包含：人物、事件标题、时间表述、地点、描述

### 阶段 4：时间解析
- dateparser/ctparse 解析时间
- 相对时间通过上下文锚点推算
- 模糊时段基于生卒年估算

### 阶段 5：多人事件对齐
- 时间窗口 + 事件类型相似度初筛
- 标题向量相似度 + 人名重叠 + 地点重合计算
- 自动对齐 / 待确认 / 独立 三档

## 7. 分阶段开发路径

| 阶段 | 目标 | 关键任务 |
|------|------|----------|
| 第一阶段 | 单人物时间轴 MVP | 文本提取、事件抽取（大模型辅助）、时间解析、单人物 CRUD、单泳道 D3 时间轴、管理端基础 |
| 第二阶段 | 多人物与共享事件 | 实体对齐、多泳道前端、对齐工作台、多人物对比视图 |
| 第三阶段 | 自动化与智能增强 | 外部信息自动收集、OCR 集成、时间校准、交互优化 |
| 第四阶段 | 丰富与打磨 | 权限路由、响应式适配、嵌入分享、性能优化 |

## 8. 开发规则

### 核心原则
1. **宁简勿繁** — 不要过度设计，能用数组不用类，能用函数不用框架。在遇到三个重复之前不做抽象
2. **约定大于配置** — 遵循项目已有模式，不引入新的架构范式
3. **先跑通再优化** — 第一阶段目标功能可用，不追求性能、扩展性、通用性
4. **每个函数都要写注释** — 说明函数用途、参数含义、返回值，方便团队成员理解

### Commit 规范
- 使用 Conventional Commits: `feat:`, `fix:`, `refactor:`, `docs:`, `chore:`
- 提交信息控制在 50 字以内，聚焦 Why 而非 What

### 前后端并行开发
- 窗口 1（后端）：FastAPI 开发服务器
- 窗口 2（前端）：Vite 开发服务器
- API 契约定义先行，前端 mock 数据并行开发

## 9. 注意事项

- 人名识别是基础，NER 阶段需用历史人物词表和白名单增强
- 时间矛盾处理：原始差异保留在 person_event 字段，标准事件取最可靠来源
- Agent 成本控制：小模型做全量初筛，大模型只做难例抽取或最终审核
- 版本与追溯：事件修改需记录版本，便于回滚
