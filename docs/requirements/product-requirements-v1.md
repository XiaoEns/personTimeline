# 多人物传记时间轴系统 — 产品需求文档

## 1. 项目概述

构建一个支持多人物传记的时间轴可视化与知识管理系统。用户可以上传人物传记文本，系统自动提取人物、事件、时间信息，处理时间点/时间段的混合表示，并能够在多人传记之间对齐共享事件。前端以泳道时间轴进行可视化展示，系统分成管理模块（数据生产）与展示模块（叙事阅读）两大部分。

## 2. 核心概念与数据结构

### 2.1 时间表达的统一模型

传记中的时间既有精确时间点（"1949年10月1日"），又有时间段（"1949-1956年"）和模糊时段（"少年时代"）。

方案：用字段组合覆盖所有形态，将"原文描述"与"可计算机时间"分离。

- 时间段：`start_date` ≠ `end_date`
- 时间点：`start_date` = `end_date`
- 模糊时段：用近似范围存储，标记 `granularity`

### 2.2 多人物共享事件模型

从单人时间轴扩展到多人，将事件提升为独立实体，通过关联表承载每个人物的视角。

三张核心表：`person`（人物）、`event`（标准事件）、`person_event`（人物-事件关联）

### 2.3 人物别名与外部信息扩展

支持人物识别与重名处理：`person_alias` 表存储别名映射，`external_person_info` 表存储自动采集的外部信息。

## 4. 事件管理策略

- **手动创建是基础能力**，用户可完全脱离 AI 使用系统
- AI 自动抽取是增强功能，抽取结果需人工审核确认
- 手动 CRUD 与 AI 抽取并存

## 5. 页面结构

### 5.1 管理端（/admin）

管理端通过左侧导航栏包含三个平级菜单：**人物管理**、**事件管理**、**上传与抽取**。

#### 5.1.1 人物管理（`/admin/persons`）

人物列表表格展示所有已创建的人物，支持分页、姓名模糊搜索、状态（草稿/已发布）筛选。

表格列：**姓名**、**生卒年**（基于规范化日期显示）、**状态**（草稿/已发布 Tag）、**事件数**（关联事件计数）、**创建时间**。

每行操作按钮：
- **编辑** → 打开 PersonFormDialog 弹出框，可修改姓名、生卒日期（含原文表述）、简介、状态，支持别名管理（添加/删除别名）
- **时间轴** → 跳转展示端 `/view/persons/:id`（单人物时间轴视图）
- **Git树** → 跳转展示端 `/view/persons/:id/git-tree`（多人物分支树视图）
- **事件** → 跳转事件管理 `/admin/events?person_id=:id`，按该人物筛选事件
- **删除** → Modal.confirm 确认后删除（级联删除所有关联数据）

新建人物通过"新建人物"按钮触发 PersonFormDialog 弹出框。

#### 5.1.2 事件管理（`/admin/events`）

独立的顶级页面，展示所有事件列表，支持分页、标题模糊搜索、事件类型筛选（出生/死亡/教育/仕途/创作/历史/其他）、时间类型筛选（时间点/时间段/模糊）、排序（可按多字段排序）。

支持从人物列表携带 `person_id` 参数跳转，进行按人物关联筛选。

每行操作按钮：**编辑**、**删除**。
新建/编辑事件 → EventFormDialog 弹出框，包含标题、描述、事件类型、时间类型、时间粒度、开始/结束日期、显示文本、地点、来源、关联人物等字段。

#### 5.1.3 上传与抽取（`/admin/upload`）

完整的数据采集与 AI 抽取流程：

1. **选择人物** → 从下拉列表选择目标人物
2. **上传传记文件** → 拖拽/点击上传 TXT 或 PDF 文件，自动提取文本并存入 `biography_text` 表
3. **传记文本管理** → 展示该人物下所有已上传的传记文本列表，支持按篇删除
4. **AI 事件抽取** → 调用 LLM 从传记文本中提取结构化事件，以 `is_inferred=true` 创建 Event 记录并自动关联到该人物；抽取结果以结构化列表展示

#### 5.1.4 对齐工作台（待实现 ⚠）

> 计划第二阶段实现：待确认的事件对齐对，并排对比、一键合并/拆分。

### 5.2 展示端（/view）

#### 5.2.1 单人物时间轴视图（`/view/persons/:id`）

以横向可缩放时间轴展示单个人物的完整事件时间线。

- **人物信息区**：顶部显示人物姓名、生卒年份、简介
- **时间轴可视化**：D3.js 绘制，支持鼠标拖拽平移、滚轮缩放（0.1x ~ 100x）、全屏模式、网格线显隐
- **事件交互**：点击时间轴上的事件节点弹出 EventCard 详情卡片
- **自适应**：根据可视范围自动调整刻度间距（年/季度/月/周/日/小时）

#### 5.2.2 多人物 Git 分支树视图（`/view/persons/:id/git-tree`）

以 Git 提交历史的可视化风格展示多个人物的时间线交叉关系。

- **主分支**：初始以一个人物为主分支，其事件按时间线排列成 commit 链
- **添加分支**：点击某事件的 EventCard 中"相关人物"按钮，可将该人物添加为新的分支（branch）加入图中
- **Merge Commit**：多人共同参与的事件表现为 merge commit，多条分支在该节点合并
- **分支管理**：支持移除分支（主分支除外），顶部显示分支图例（名称、加载状态、移除按钮）和统计信息（分支数、事件数）
- **图渲染**：使用 Git 图渲染引擎（Graph + Table 模块），浅色主题，彩虹色分支

#### 5.2.3 事件详情浮层卡片（EventCard）

模态浮层展示事件详细信息，包含：事件标题（优先显示个人视角标题）、时间范围、事件类型、角色。相关人物列表支持两段式点击交互：首次点击变为"确认添加"态，再次点击确认将该人物加入 Git 树分支。

## 6. 数据处理管道

### 阶段 1：文本提取 ✅ 已实现
- TXT 直接读取（UTF-8），PDF 用 PyMuPDF 逐页提取文本
- 原始文本存入 `biography_text` 表
- API：`POST /api/upload`（上传文件）、`GET /api/persons/{id}/biography`（查询传记列表）、`DELETE /api/biography/{id}`（删除传记）

### 阶段 2：NER 与指代消解 ❌ 待实现
> 计划第二阶段实现
- spaCy/HanLP + 历史人物词表增强
- 通过 `person_alias` 表映射到标准 `person_id`

### 阶段 3：事件抽取 ✅ 已实现
- 大模型结构化抽取，通过 OpenAI 兼容 API 调用（支持 DeepSeek 等），返回 JSON
- 包含：事件标题、描述、时间表述、时间类型、粒度、事件类型、地点
- 抽取结果以 `is_inferred=true` 创建 Event 记录并自动关联到目标人物
- API：`POST /api/persons/{id}/extract`

### 阶段 4：时间解析 ⚠️ 部分实现
- 当前方案：LLM 内置时间解析能力 + 前端表单手动输入规范化日期
- dateparser/ctparse 库尚未集成，计划第二阶段引入用于增强解析准确率
- 相对时间通过上下文锚点推算（待实现）
- 模糊时段基于生卒年估算（待实现）

### 阶段 5：多人事件对齐 ❌ 待实现
> 计划第二阶段实现
- 时间窗口 + 事件类型相似度初筛
- 标题向量相似度 + 人名重叠 + 地点重合计算
- 自动对齐 / 待确认 / 独立 三档

## 8. 开发规则

### 核心原则
1. **宁简勿繁** — 不要过度设计，能用数组不用类，能用函数不用框架。在遇到三个重复之前不做抽象
2. **约定大于配置** — 遵循项目已有模式，不引入新的架构范式
3. **先跑通再优化** — 第一阶段目标功能可用，不追求性能、扩展性、通用性
4. **每个函数都要写注释** — 说明函数用途、参数含义、返回值，方便团队成员理解

### Commit 规范
- 使用 Conventional Commits: `feat:`, `fix:`, `refactor:`, `docs:`, `chore:`
- 提交信息控制在 50 字以内，聚焦 Why 而非 What

### 前后端并行开发
- 窗口 1（后端）：FastAPI 开发服务器
- 窗口 2（前端）：Vite 开发服务器（React）
- API 契约定义先行，前端 mock 数据并行开发

## 9. 注意事项

- 人名识别是基础，NER 阶段需用历史人物词表和白名单增强
- 时间矛盾处理：原始差异保留在 person_event 字段，标准事件取最可靠来源
- Agent 成本控制：小模型做全量初筛，大模型只做难例抽取或最终审核
- 版本与追溯：事件修改需记录版本，便于回滚
